# パッケージ同梱アドオン 手動検証チェックリスト

自動テスト（Unit / 統合）で検証できない、実機・実 WebView 依存の受け入れ基準（AC 4.5 / NFR-004）を人手で確認するための手順です。

- 対象: Tauri デスクトップアプリ（`npm run tauri:dev` または `npm run tauri:build` のバンドル）
- 前提: 実 WebView 上での `asset://` スクリプト実行（Issue #5 PoC で macOS は確認済み）

## 準備: 同梱アドオン付き `.tgz` を 2 つ作る

```bash
# アドオンをビルドして同梱パッケージを作成（A・B の 2 種）
npm run build:addons
npm run export:slides -- --name pkgA --slides slides.json --addons
npm run export:slides -- --name pkgB --slides slides.json --addons
# → dist-slides/slides-pkgA-1.0.0.tgz, slides-pkgB-1.0.0.tgz
```

> A と B で異なるコンポーネント名を使うと残留・混線の確認がしやすい（`addons/src/*/entry.ts` を編集して別名で登録した B を用意する）。

```bash
npm run tauri:dev
```

## チェック項目

### AC-1: 同梱アドオンのランタイムロード（FR-001）
- [x] pkgA を開く → 初回確認ダイアログで「有効化する」を選ぶ（2026-07-23 実機確認）
- [x] スライドの `{ "component": { "name": ... } }` が正しく描画される（`Component not found` にならない）（2026-07-23 実機確認）

### AC-2: A→B→A 切替で残留・混線なし（FR-003 / NFR-003）
- [ ] pkgA→pkgB→pkgA と切り替える
- [ ] 各切替後、そのパッケージのコンポーネントが正しく描画される
- [ ] 前のパッケージ固有コンポーネントが残っていない（B 表示中に A のものが出ない）

### AC-3: 再オープンで二重注入なし（FR-004）
- [ ] 同じ pkgA を続けて 2 回開く
- [ ] DevTools の Elements で `<head>` 内の同一 `data-addon-src` の `<script>` が 1 個だけであること

### AC-4: 発表者ビューでの解決（FR-006）
- [ ] pkgA を開いた状態で発表者ビューを開く
- [ ] 発表者ビュー側でも同梱コンポーネントが描画される（fallback にならない）
- [ ] 発表者ビューを開いたまま pkgB に切り替え → 発表者ビューも B の内容に追従する

### AC-5: セキュリティ（既定拒否・オプトアウト）（FR-008/009 / NFR-001）
- [x] 未判断のパッケージ初回オープンで確認ダイアログが出る（2026-07-23 実機確認）
- [ ] 「無効のまま」を選ぶとアドオンがロードされず、スライドは fallback 付きで開ける
- [ ] 一度判断したパッケージは再オープン時に再確認されない
- [ ] 設定で「同梱アドオンを常に無効化する」を ON にすると、確認ダイアログなしでアドオンが無効になる
- [ ] 設定の「アドオンの許可履歴をリセット」後、再度確認ダイアログが出る

### AC-6: フォールバックファースト（A-005）
- [ ] 壊れた/存在しないアドオンを含むパッケージでも、スライド自体は開ける（該当コンポーネントのみ fallback 表示）

### AC-7: ホーム復帰・サンプル表示でのクリア（NFR-003）
- [ ] パッケージ表示中にホームへ戻り、サンプルスライドを開く
- [ ] 前パッケージの custom コンポーネントが残っていない

### NFR-004: Windows(WebView2) 実機
- [ ] Windows 環境で AC-1〜AC-6 を再確認する（`http://asset.localhost/…` 形式での `<script>` 実行）
